## Findings

**HIGH**

1. **[load.go, `validateAlias`]** The alias-key string is passed straight into `validateCharset` without an emptiness guard, and `validateCharset`'s `for i, r := range s` loop vacuously returns `nil` on `s == ""`; a `models.yaml` entry keyed `""` (e.g. `"": {engine: claude, model: x}`) is therefore silently accepted into the `Registry`, even though the file's own docstring (load.go:30) states "an alias key must match `[a-z0-9-]+`" (one-or-more chars). — HIGH

2. **[registry.go, `Registry.Resolve`]** `Resolve`'s form-dispatch is just `if s.Alias == ""`, which is also true for a totally zero-value `Spec{}` — a shape `Parse` can never itself produce (it rejects empty alias). Calling `Resolve(Spec{})` returns `Resolved{Engine:"", Model:"", Params:{}}, nil` with no error, silently manufacturing a garbage empty-engine/empty-model result instead of failing loudly. — HIGH

**MEDIUM**

3. **[load.go, `LoadRegistry`]** Duplicate top-level alias keys in the YAML document (two `sonnet:` blocks in the same file) are not detected — `yaml.v3`'s default map-decoding silently keeps only the last occurrence — which is inconsistent with `parseBracket`'s explicit duplicate-key rejection for `[k=v,...]` and could mask an operator's copy-paste mistake. — MED

4. **[registry.go, `Registry.Resolve`]** Resolve performs no validation on a hand-constructed `Spec`/`Registry` that bypassed `Parse`/`LoadRegistry` — e.g. `Spec{Engine:"gemini"}` or a `Registry` entry with an unknown engine — so the `knownEngines`/`knownParams` closed-vocabulary guarantee the package advertises is only enforced at the two entry points, not at the point that actually matters (Resolve), even though `Spec`, `Entry`, and `Registry` are all exported, mutable types callers can construct directly. — MED

5. **[registry.go, `Registry.Resolve`]** Resolve doesn't check that exactly one of (Alias) / (Engine+Model) is set; a malformed `Spec{Alias:"sonnet", Engine:"claude", Model:"x"}` built by hand is treated as pure alias form and silently discards `Engine`/`Model` with no error, rather than rejecting the "both shapes set" contradiction the package's own doc comment says can never hold. — MED

6. **[design, alias vs. escape form]** `claude:sonnet` (escape form) and `sonnet` (alias form) both parse and resolve without error but produce materially different results — escape form never consults registry `Defaults` — so a plausible operator typo (adding a stray `claude:` prefix) silently drops an intended default (e.g. opus's `effort: high`) with zero diagnostic. — MED

**LOW**

7. **[parse.go, bracket parsing]** Inputs with a well-terminated-but-malformed bracket structure, e.g. `sonnet[effort=high]evil]` or `sonnet[a=1][b=2]`, aren't rejected with a clear structural error; the extra `]`/second bracket gets folded into `bracketInner` and only surfaces later as a confusing "invalid character ']'" error on a param value rather than a "malformed bracket" error. — LOW

8. **[parse.go/modelspec.go, `isModelIDChar`]** Escape-form model-id and param-value charset excludes uppercase letters and characters like `/`, `@`, `+` common in real provider model identifiers; only "claude" is a known engine today so this doesn't bite yet, but the grammar has no escape hatch, so a future engine with such IDs would require a parser change. — LOW

9. **[load.go, `LoadRegistry`]** `yaml.Decoder.Decode` reads only the first document in the YAML stream; a `models.yaml` accidentally containing a second `---`-delimited document has that second document silently ignored with no error. — LOW

10. **[leaf_enforcement_test.go, `TestLeafInvariant_AllowlistOnly`]** A production file that fails `parser.ParseFile` is logged via `t.Logf` and treated as passing (`return nil`, never added to `failures`) instead of failing the test — inconsistent with the package's fail-loud philosophy, though a real syntax error would already fail compilation first, limiting practical impact. — LOW

11. **[registry.go, unknown-alias error]** The unknown-alias error unconditionally lists every known alias via `sortedKeys(r)` with no cap; harmless today with 4–5 builtins, but unbounded if a hub's `models.yaml` grows large. — LOW

12. **[test coverage]** No test exercises: an empty-string alias key (finding 1), a zero-value `Spec{}` passed to `Resolve` (finding 2), duplicate top-level alias keys (finding 3), a hand-built `Spec` with both alias- and escape-form fields set (finding 5), or multi-document YAML (finding 9) — the gaps that let findings 1, 2, 3, 5, and 9 go undetected. — LOW/MED (meta-finding)
