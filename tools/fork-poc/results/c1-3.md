## Findings

**parse.go**

1. [parse.go, `Parse` escape-form model-id/param charset errors] Position values reported in "invalid character ... at position N" are computed relative to the sliced substring (`model`, `key`, `value`), not the original spec string `s`, so for escape-form model IDs and all bracket params the reported position is wrong/misleading relative to what the operator actually typed — MED.

2. [parse.go, `Parse` bracket detection via first `[` + suffix `]`] When trailing garbage after a legitimate bracket happens to end in `]` (e.g. `sonnet[effort=high]x]`), the code silently reinterprets everything up to the last `]` as bracket content instead of reporting "must end with ']'", producing a confusing/wrong error (e.g. "invalid character in param value") instead of the intended trailing-text error — LOW.

3. [parse.go, `validateCharset`] Position index is a byte offset (from `range` over the string), not a rune offset; for any future non-ASCII input this would misreport the visual character position — LOW (currently unreachable since whitespace check runs first and would likely fire first, but it's a latent inconsistency).

**registry.go**

4. [registry.go, `Registry.Resolve`] Resolve trusts `s.Alias == ""` as the sole discriminant between alias-form and escape-form and never validates that the "other side" of the Spec is well-formed: a zero-value `Spec{}` silently resolves to `Resolved{Engine:"", Model:"", Params:{}}` with no error, and a malformed dual-shape `Spec` (both Alias and Engine/Model set) silently resolves as alias-form while dropping Engine/Model — since `Spec`'s fields are exported and not constructor-guarded, this breaks the package's own "fail loud, never silently tolerate malformed input" philosophy stated in parse.go — HIGH.

5. [registry.go, `Resolve` unknown-alias error] When called against a nil/zero-value Registry, the error's "known aliases" list is empty (`[]`), which is technically correct but unhelpful/confusing UX — LOW.

**load.go**

6. [load.go, `validateAlias`] Unlike `Parse`, which explicitly rejects an empty alias (`body == ""`), `validateAlias` only calls `validateCharset("", ...)`, which vacuously succeeds on an empty string — a models.yaml with an empty-string alias key (`"": {engine: claude, model: x}`) is accepted by `LoadRegistry` even though that alias can never be referenced by any string `Parse` would accept, leaving dead/unreachable registry state with no error — MED.

7. [load.go, `validateAlias`] `Entry.Model` is only checked for non-empty; it is never run through `isModelIDChar`/charset validation the way an escape-form model-id from `Parse` is, so a models.yaml can inject a `Model` string containing arbitrary characters (spaces, brackets, control characters) that would be rejected if typed directly in a spec string — inconsistent validation between the two entry points for the same data, and such a string flows unsanitized into `Resolved.Model` for downstream consumers — MED.

8. [load.go, `validateAlias`] `Entry.Defaults` values are checked only for non-empty, never charset-validated, so a models.yaml can set a default param value containing characters Parse would reject in a bracket (`effort: "high]weird"`), again bypassing the charset discipline applied elsewhere and reaching `Resolved.Params` unsanitized — MED.

9. [load.go, `LoadRegistry`] YAML is decoded with a single `Decode` call into `map[string]Entry`; duplicate alias keys within one models.yaml document depend entirely on yaml.v3's undocumented-in-this-package default duplicate-key behavior rather than an explicit loud check, so it's unclear (and untested) whether a hand-edited file with the same alias block repeated twice is rejected or silently last-wins — LOW/MED (uncertain, worth an explicit test/check given the package's stated "fail loud" contract).

10. [load.go, `LoadRegistry`] Only the first YAML document is decoded; a models.yaml containing a `---` document separator with content after it is silently ignored rather than flagged as malformed — LOW.

**modelspec.go**

11. [modelspec.go, `Spec`/`Entry`/`Registry`] All fields are exported with no constructor or invariant-checking method, so nothing in the type system or at construction time prevents building a `Spec` that violates the documented "exactly one of two shapes" invariant — the whole package's correctness for downstream callers rests on every caller happening to only ever pass `Parse`'s output into `Resolve`, which is not enforced (root cause of finding 4) — MED.

12. [modelspec.go, `knownParams`/`knownEngines`] These closed-vocabulary sets are mutable package-level `var` maps rather than something read-only/immutable; nothing in this package prevents accidental mutation from other code in the same package (e.g. a future helper or test) silently expanding the vocabulary at runtime — LOW.

**leaf_enforcement_test.go**

13. [leaf_enforcement_test.go, `TestLeafInvariant_AllowlistOnly`] The stdlib heuristic (`no '.' in first path segment ⇒ stdlib`) will misclassify any non-stdlib import path that happens to lack a dot in its first segment (e.g. a local/relative module path or an internal monorepo alias) as stdlib, silently passing it through the allowlist check instead of flagging it — a false-negative gap in the enforcement mechanism itself — LOW/MED.

14. [leaf_enforcement_test.go, `TestLeafInvariant_AllowlistOnly`] When `parser.ParseFile` fails on a production file, the test only does `t.Logf` and continues (`return nil`) rather than calling `t.Error`/`t.Fatal` — a production file that fails to parse silently skips the leaf-invariant check instead of failing loudly, inconsistent with the rest of the package's fail-loud philosophy — LOW.

**Test coverage gaps**

15. [registry_test.go] No test exercises `Resolve` with a malformed/dual-shape `Spec` (both Alias and Engine/Model set, or a genuine zero-value `Spec{}`) — the exact gap identified in finding 4 is untested — MED (test gap tied to a HIGH correctness gap).

16. [load_test.go, `TestLoadRegistry_RejectsInvalidEntries`] The table doesn't include a case for an invalid/uppercase alias key or an empty-string alias key in the YAML file, so the charset validation path for aliases (and the gap in finding 6) is untested — MED.

17. [load_test.go / parse_test.go] No fuzz test (`go test -fuzz`) exists for `Parse`, despite it being a hand-written strict grammar over untrusted/hand-typed operator input — a natural candidate for fuzzing to catch panics or charset/position edge cases (e.g. multi-byte runes) that table tests won't surface — LOW.

18. [load_test.go] No test covers a models.yaml with a duplicate top-level alias key (ties to finding 9) — LOW/MED.

19. [template.yaml / template_test.go] The seed template's correctness is enforced only by a runtime test that loads it through `LoadRegistry`; there is no static/compile-time guarantee, so a hand-edit to `template.yaml` that breaks the seed (e.g. adds an undocumented alias or bad charset) is only caught if `go test` is run, not at build time — LOW (inherent to the embed-a-string-template approach, but worth noting as a risk).
