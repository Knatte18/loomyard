## Findings

1. **[registry.go, `Registry.Resolve`]** Resolve never validates that a `Spec` satisfies the package's core "exactly one shape" invariant — a hand-built `Spec{}` zero value, or a `Spec` with both `Alias` and `Engine`/`Model` set, is not rejected: the check `if s.Alias == ""` silently routes to escape form for the former (producing a `Resolved{Engine:"", Model:"", Params:{}}` with no error) and to alias form for the latter (silently discarding whatever `Engine`/`Model` were set, with no warning). Since `Spec`'s fields are all exported, any caller constructing one directly (not just via `Parse`) can trigger this. — **HIGH**

2. **[registry.go/load.go, `Resolve` & `validateAlias`]** The closed-vocabulary/shape invariants (`knownEngines`, `knownParams`, "exactly one shape") are enforced only at the two entry points (`Parse`, `LoadRegistry`'s `validateAlias`), not on the underlying types themselves. Any code that builds a `Registry{...}` or `Entry{...}` literal directly (several unit tests already do this) bypasses all validation — e.g. an `Entry{Engine: "gemini"}` built by hand resolves without error. — **MED**

3. **[parse.go, `parseBracket`/`knownParams`]** Param *keys* are validated against a closed set (`effort`, `version`), but param *values* are not — any string matching `isModelIDChar` is accepted for `effort` (e.g. `effort=hgih` or `effort=zzz` parses successfully), undermining the package's stated "fail loud" philosophy; a typo'd effort level is only caught (if at all) deep in a downstream consumer. — **MED**

4. **[parse.go, error messages]** "unknown param key" and "unknown engine" errors don't enumerate the valid options, unlike the unknown-alias error (which lists `sortedKeys`). Inconsistent operator-facing diagnostics. — **LOW/MED**

5. **[modelspec.go, `knownParams`/`knownEngines`]** These closed vocabularies are mutable package-level `var` maps with no synchronization or defensive-copy protection. Any accidental mutation (e.g. from a misbehaving test in the same process) would silently corrupt validation for every subsequent `Parse`/`Resolve` call package-wide. — **LOW**

6. **[registry.go, `Registry`]** No documented (or enforced) concurrency contract: `Registry` is a bare map type, so concurrent `Resolve` reads against a `Registry` that another goroutine is mutating (e.g. adding aliases) is a data race with no guard or doc warning. — **LOW**

7. **[template.yaml vs registry.go `builtins()`]** The shipped seed template's effort defaults (`sonnet`→medium, `opus`→high, `fable`→high) have no structural/programmatic link to `builtins()` (which carries zero defaults for every alias). If a hub never materializes `models.yaml` (e.g. seeding is skipped or fails), every consumer silently gets no effort defaults at all instead of the documented ones, and nothing catches future drift between the hardcoded template values and `builtins()`. — **MED**

8. **[leaf_enforcement_test.go, `TestLeafInvariant_AllowlistOnly`]** A parse failure on a production `.go` file is only logged (`t.Logf`) and the file is skipped rather than failing the test outright — meaning an unparsable file (however unlikely, since the package must compile for the test to run at all) could silently escape the banned-import check. Safety-net gap, low practical likelihood. — **LOW**

9. **[parse.go, `Parse`]** Inputs with multiple `[`-like sections (e.g. `sonnet[a=1][b=2]`) are still correctly rejected, but via a misleading "invalid character" charset error on the resulting garbled value rather than a clear "multiple brackets" or "trailing content" message — correctness is fine, diagnostics are confusing. — **LOW**

10. **[load_test.go, test gap]** No test exercises an invalid alias charset in `models.yaml` (e.g. an uppercase or symbol-containing alias key), even though `validateAlias` explicitly implements and depends on this check via `validateCharset`. — **MED**

11. **[load_test.go, test gap]** No test covers the "file exists but is unreadable / is a directory" branch of `LoadRegistry` (the wrapped non-`ErrNotExist` read-error path) — only the absent-file and malformed-content paths are tested. — **LOW**

12. **[parse_test.go, test gap]** No fuzz/property testing of `Parse` despite it being a hand-rolled, operator-string-facing strict grammar parser; nothing asserts it never panics on adversarial input (multi-byte Unicode, deeply nested/garbled brackets, etc.), relying entirely on the hand-picked table cases. — **LOW**

13. **[modelspec.go, `Resolved.Params` design]** A consumer reading `resolved.Params["effort"]` for a key that's genuinely absent gets a silent `""` (Go map zero value) rather than any distinguishable "not set" signal — risks a downstream consumer treating empty string as a valid/default effort rather than "unspecified." — **LOW**

14. **[registry.go, `Resolve` alias-form path]** Beyond invariant-bypass (#2), `Resolve` also doesn't sanity-check the looked-up `Entry` itself (e.g. an `Entry` with empty `Model` or empty `Engine` slipped in via a hand-built `Registry`) before returning it as a "fully resolved" `Resolved` — the function's contract implies a valid result but doesn't defensively guarantee one. — **LOW**
