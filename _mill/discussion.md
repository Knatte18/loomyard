## Problem

The weft model (proposal `proposal-weft-repo.md`, roadmap milestone 7 / "Task 008")
splits lyx artifacts out of the host repo into a companion **weft** repo, reached through
junctions. The **weft engine** (paths geometry, paired `lyx worktree add` spawn, and
`lyx weft status|commit|push|pull|sync`) already landed (task 006). This task is the
**producer side**: give operators a first-class way to view and edit module configuration
(`lyx config`), and make the surrounding infrastructure **extensible** so that a future
`codeguide` module can be dropped in with near-zero rework.

**Why now:** config files already moved to `_lyx/config/<module>.yaml` (the loader, init
scaffolding, and tests all use that layout — see Technical context), but there is **no
command to edit them**: today an operator must hand-edit YAML inside the weft worktree and
remember to run `lyx weft sync`. And the worktree junction-seeder hardcodes `_lyx`, so the
layout knowledge that should live in `internal/paths` (the sole geometry owner) is
partially scattered. Both gaps block the codeguide work that comes next.

## Scope

**In:**

- A `lyx config` command:
  - **Bare `lyx config`** → interactive numbered top-menu listing the editable modules
    (mirrors the `lyx ide menu` stdlib pattern), then edits the chosen one.
  - **`lyx config <module>`** → edits that module's config directly.
  - Edit flow: resolve `_lyx/config/<module>.yaml` via the host relpath (the junction
    routes the bytes into the weft worktree); **scaffold from the module's commented
    template** if the file is missing; open in `$EDITOR`; **validate** (re-parse YAML);
    then trigger **`lyx weft sync`**.
- Centralize the load/scaffold/edit/validate **machinery** in `internal/config`
  (no weft dependency — see Constraints/layering); the **edit+sync composition** and the
  command surface live one level up in a new CLI layer that may import both `config` and
  `weft`.
- Relocate the per-module **commented-template generators** out of `internal/board/init.go`
  into each owning module (`board`, `worktree`, `weft`), so each module owns its schema +
  template; `init` reuses the same generators (single source of truth).
- **Generalize the worktree junction-seeder** to iterate a junction list **owned by
  `internal/paths`** instead of hardcoding `_lyx`. The list has exactly **one entry
  (`_lyx`)** today; this is purely a mechanism generalization so a future `_codeguide`
  entry is a one-line addition.
- Tests via the existing `internal/lyxtest` programmatic paired host+weft fixtures
  (`integration` build tag) for the seeder/sync paths, plus unit tests for the `lyx config`
  machinery using an injected fake editor.

**Out:**

- **Anything codeguide-specific.** No `_codeguide` junction seeding, no `_codeguide` in any
  weft pathspec, no `HostCodeguideLink` method, no `codeguide.yaml` schema, no `codeguide`
  entry in the junction list or the config-module registry. The codeguide module **does not
  exist yet and is not designed here**; it is a separate, later task. This task only makes
  the *general* mechanisms extensible enough that codeguide is a clean drop-in.
- The `_lyx/config/` directory migration — **already done** (see Technical context).
- The `loom-git-clone` hub-creator (task 007) and any hub-creation/validation logic. The
  full "hub layout spec" (origins, branches, board repo) is the hub-creator's concern.
- Editing the external `codeguide:*` plugin skills.
- Fixing or depending on `C:\Code\LyxTestHub` — it is built with the **stale** `_weft/<slug>`
  nested layout and is **not** the test substrate; it will be regenerated by `loom-git-clone`.
- Portals (remain on hold).

## Decisions

### config-command-surface

- Decision: `lyx config <module>` (module as **argument**), plus a bare `lyx config`
  interactive top-menu that lists modules and dispatches into the same edit flow.
- Rationale: a single config-CLI owns the **edit-then-`weft sync`** composition and is the
  one place allowed to import both `internal/config` and `internal/weft`. The top menu has
  a natural home (bare `lyx config`). Module ownership is preserved because each module
  still owns its schema + commented template; only the machinery is centralized.
- Rejected:
  - `lyx <module> config` (per-module subcommand) — would force `board`/`worktree` to
    import `weft` to trigger the sync (new cross-module coupling), and leaves no clean home
    for a bare-`lyx config` menu.
  - Top-level interactive menu only (no `<module>` arg) — operator found the
    pick-from-menu-every-time flow cumbersome; the direct `lyx config <module>` form is
    wanted alongside the menu.

### centralized-edit-machinery-in-config

- Decision: `internal/config` gains the **pure** edit machinery — resolve path, scaffold
  from a passed-in template if missing, open `$EDITOR`, re-parse to validate. It does **not**
  call `weft`. The edit+sync composition lives in the new CLI layer above.
- Rationale: `internal/weft` already imports `internal/config` (weft loads its own config
  via `config.Load`). Putting a `weft sync` call inside `internal/config` would create a
  circular dependency. Keeping `config` weft-agnostic preserves the layering.
- Rejected: a `config.RunCLI` in `internal/config` that calls weft directly — circular import.

### module-owned-templates

- Decision: each module exposes its own commented-config-template generator
  (`board`, `worktree`, `weft`); `init` and `lyx config` both consume these.
  - **`board` and `worktree`:** **relocate** the existing `generateCommentedBoardYAML` /
    `generateCommentedWorktreeYAML` out of `internal/board/init.go` into the respective
    modules. These are moves — `init` must scaffold byte-identical content afterward
    (regression guard, see Testing).
  - **`weft`:** the commented-template generator is **authored fresh** — `weft` today has only
    `DefaultConfig()` returning a struct (`internal/weft/config.go:22`), no commented YAML. Its
    single commented key is `pathspec` (default `_lyx`). The "identical-content" regression
    guard therefore applies **only to board/worktree**, not weft.
- Rationale: "modules own their configs" (operator's explicit steer). Single source of
  truth for each module's default/commented YAML, reused by both scaffolding paths.
- Rejected: keeping templates in `board/init.go` and importing from there — spreads schema
  knowledge into the wrong module and couples the config-CLI to `board`'s init internals.

### junction-list-owned-by-paths

- Decision: add a centralized host-junction list to `internal/paths` (e.g.
  `HostJunctions()` returning `{name, link-path, weft-target}` records) with **`_lyx` as the
  sole entry**. The `internal/worktree` seeder (`seedLyxJunction` + the `.git/info/exclude`
  seeding) iterates this list instead of hardcoding `_lyx`. The record carries **three fields
  because the two seeding ops consume different ones**: junction creation (`seedLyxJunction`)
  uses **link-path + weft-target**, while exclude seeding (`seedGitExclude`,
  `internal/worktree/weft.go:137`, which appends the literal directory name and does a
  line-exact match) iterates the **`name`** field. The record shape must cover both.
- Rationale: `internal/paths` is the sole geometry owner (CONSTRAINTS.md). The seeder
  hardcoding `_lyx` leaks layout knowledge. Iterating a paths-owned list centralizes the
  "what junctions a host worktree has" fact and makes a future `_codeguide` entry a one-line
  addition — the extensibility the operator asked for, with **no codeguide code today**.
- Rejected:
  - Adding `HostCodeguideLink` now — operator explicitly scoped codeguide out ("zero
    codeguide geometry"); even pre-staging the geometry method is deferred to the codeguide
    task.
  - Leaving the seeder hardcoded — fails the geometry-owner invariant and gives the future
    codeguide task more to change.

### config-module-registry

- Decision: the config-CLI holds a registry mapping module name → its commented-template
  generator, currently `{board, worktree, weft}`. Bare `lyx config` lists the registry keys
  (optionally marking which already have a file); `lyx config <module>` looks the module up.
  A future codeguide module is added by registering one entry — no command-logic change.
- Rationale: generic enough that codeguide slots in later; explicit enough to scaffold a
  module whose file does not exist yet (which a "list existing `*.yaml`" approach cannot do).
- Rejected: dynamically enumerating existing `_lyx/config/*.yaml` only (can't offer to
  create a missing module's file); hardcoding the module names inside the command body.

### edit-validate-and-sync-behavior

- Decision: after `$EDITOR` exits, re-parse the YAML. Loop and abort semantics are **explicit**:
  - **Abort triggers** (deterministic, so the fake-editor tests can drive them): (a) the editor
    process exits **non-zero**; or (b) after a validation failure the file content is
    **unchanged** from the previous failed attempt (operator saved without fixing, or exited
    without editing). On abort: leave the file as-is, **skip sync**, print a human-readable
    message, and return **exit code 1**.
  - **Re-edit loop:** on a parse failure where the content *did* change, print the parse error
    and **re-open the editor**.
  - **Success:** on valid YAML, trigger `lyx weft sync` by calling `weft.RunCLI` with its output
    routed to **`io.Discard`** (NOT the interactive `out`) — `weft sync` emits `{"ok":true}`
    JSON (`internal/weft/cli.go:147`), which must not contaminate `lyx config`'s human-readable
    stream. The config-CLI inspects the returned exit code and prints its **own** human-readable
    confirmation ("synced") or, on non-zero, a human-readable sync error. `weft sync` is a no-op
    when nothing changed, so an unchanged-but-valid edit is harmless.
  - `lyx config` is interactive: it prints human-readable text and returns exit 0 on success
    (the one allowed exception to JSON output, matching `lyx ide menu`). This holds precisely
    because the JSON-emitting sync call is given a discarded writer.
  - **Push completion is NOT surfaced.** `weft sync` commits then calls `spawnPush` — a
    **detached** child (`internal/weft/cli.go:144`). So the exit code from `weft.RunCLI(["sync"])`
    reflects only **commit + push-spawn launch**, never the push's eventual result; a push that
    fails inside the detached child cannot be reported by `lyx config`. This is intentional and
    consistent with existing `lyx weft sync` semantics — the config-CLI's confirmation means
    "committed and push launched," not "pushed." The plan must not promise push-completion
    reporting.
- Rationale: prevents committing broken config; reuses the existing sync path without leaking its
  JSON into the interactive stream; gives the plan writer and tests a deterministic abort contract.
- Rejected: aborting outright on first invalid YAML (worse UX); passing the interactive `out` to
  `weft.RunCLI` (leaks JSON); an interactive y/n abort prompt (harder to drive deterministically
  in tests than the editor-exit-code / unchanged-content signals).

### editor-resolution

- Decision: use `$VISUAL` then `$EDITOR`; if neither is set, fall back to a platform default
  (`notepad` on Windows, `vi` elsewhere). The editor command is injectable so tests can
  substitute a scripted fake editor that mutates the file deterministically.
- Rationale: standard convention; testability without a real interactive editor.
- Rejected: hardcoding an editor; erroring when `$EDITOR` is unset (poor first-run UX).

## Technical context

Key files and current state (from codebase exploration):

- **Config migration is already done.** `internal/config/config.go:70` reads
  `filepath.Join(baseDir, "_lyx", "config", module+".yaml")` — the only path it looks at.
  `internal/board/init.go` scaffolds `_lyx/config/{board,worktree}.yaml`.
  `internal/config/config_test.go:TestLoad_OldFlatPathNotPickedUp` guards against the old
  flat `_lyx/board.yaml` path. So this task does **not** move directories.
- **`internal/config`** (pure library; no `RunCLI` today):
  - `FindBaseDir(cwd) (string, error)` — `config.go:31` — strict `<cwd>/_lyx` existence check.
  - `Load(baseDir, module, defaults) (map[string]string, error)` — `config.go:50` — the single
    entry point; merges defaults ← `_lyx/config/<module>.yaml`, expands `$env:` tokens.
  - Per-module typed wrappers delegate to `Load`: `board.LoadConfig` (module `board`: `path,
    home, sidebar, proposal_prefix`), `worktree.LoadConfig` (module `worktree`:
    `branch_prefix`), `weft.LoadConfig` (module `weft`: `pathspec`, default `_lyx`).
- **`internal/weft`** — full git engine already exists:
  - `RunCLI(out, args) int` — `cli.go:50` — subcommands `status|commit|push|pull|sync`.
    `sync` = `Commit(...)` then detached `spawnPush(...)` (`cli.go:139`). Reuse
    `weft.RunCLI(out, []string{"sync"})` for the post-edit sync.
  - Pathspec scoping: `scopedPathspec(relPath, cfg.Dirs())` (`weft.go:22`); `weft.Config.Pathspec`
    default `_lyx` (`config.go:24`); `Dirs()` splits on whitespace. (Already accepts
    `_lyx _codeguide`, but we do **not** change the default in this task.)
- **`internal/paths`** (sole geometry owner; CONSTRAINTS.md enforces it):
  - `Resolve(cwd) (*Layout, error)`; existing host/weft geometry methods incl.
    `HostLyxLink(slug)` (`paths.go:273`), `HostLyxLinkHere()` (`paths.go:284`),
    `WeftLyxDir()`, `WeftWorktree()`, `WeftCodeguideDir()` (`paths.go:263`, pure geometry,
    already present — but **not** consumed by this task).
  - `internal/paths/codeguide_guard_test.go` asserts no production file in `internal/paths`
    (except `paths.go`) names `_codeguide`; do not break this guard.
- **`internal/worktree/weft.go`** — `seedLyxJunction` (`:87`) creates the `_lyx` junction;
  `seedGitExclude` (`:137`) appends literal `_lyx` to `.git/info/exclude`. These are what
  the junction-list generalization refactors to iterate `paths.HostJunctions()`.
- **TUI precedent — `lyx ide menu`:** `internal/ide/menu.go:37` `Menu(l, in, out)`. Pure
  stdlib: numbered list via `fmt.Fprintf`, one line read via `bufio.NewReader(in).ReadString`,
  `strconv.Atoi`, `q` to quit, range validation. **No external TUI library** (`go.mod` has
  only `gofrs/flock`, `golang.org/x/sys`, `yaml.v3`). Clone this pattern for the `lyx config`
  top menu; inject `in`/`out` for testability.
- **Module dispatch:** `cmd/lyx/main.go:50` thin `switch module` router; add a
  `case "config":` that routes to the new config-CLI layer. The new layer is the place that
  imports both `internal/config` and `internal/weft` (and the modules, for their templates).
  **Binding constraints on the config-CLI layer** (the package *name* is mill-plan's call, but
  these requirements are not negotiable):
  - It must **not** live inside `internal/config` — `internal/weft` already imports
    `internal/config`, so a `weft` call from `config` would be a circular import.
  - It must **not** fatten `cmd/lyx/main.go` beyond the thin `case "config":` dispatch — the
    registry, menu, and edit+sync composition belong in the new layer, matching the existing
    per-module `RunCLI` pattern.
  - It is the **only** layer permitted to import both `internal/config` and `internal/weft`
    (plus the modules for their templates).
  A new `internal/configcli` package satisfies all three; mill-plan may choose another name
  that does.
- **git-ownership prototype:** `internal/board/sync.go` + `internal/git.RunGit(args, cwd)`
  (local-commit + detached coalesced-push); `weft`'s `sync.go` is the direct analog. Writes
  go through the junction (host relpath); only git is geometry-scoped and lyx-owned.

## Constraints

- **Path Invariant (CONSTRAINTS.md):** all cwd/worktree-root/weft geometry MUST go through
  `internal/paths` (`Getwd`, `Resolve`, `Layout` methods). Raw `os.Getwd` / `git rev-parse
  --show-toplevel` are banned outside `internal/paths` and `cmd/lyx/main.go` and enforced by
  `internal/paths/enforcement_test.go`. The junction-list lives in `paths` for this reason.
- **Layering / no circular imports:** `internal/weft` imports `internal/config`; therefore
  `internal/config` must remain weft-agnostic. The edit+sync composition lives above both.
- **Geometry-only codeguide:** `internal/paths/codeguide_guard_test.go` forbids new
  `_codeguide` references in `internal/paths` production files (beyond `paths.go`). This task
  adds none.
- **Git-ownership contract:** content writes go through the junction at the normal relpath;
  only git operations are lyx-owned and geometry-scoped. `lyx config` writes to the host
  `_lyx/config/...` relpath and lets `lyx weft sync` own the commit/push.
- **Documentation lifecycle (docs/overview.md):** mechanical per-module docs are deletable;
  durable design docs are kept. Update `docs/roadmap.md` (milestone 7) and `docs/overview.md`
  status notes to reflect what shipped.

## Testing

Substrate: **`internal/lyxtest`** programmatic paired host+weft fixtures
(`CopyHostHub`/`CopyPaired`/`CopyWeft`), which build real git repos in temp dirs from
`paths` geometry. Integration tests are gated behind the `integration` build tag (existing
convention; see `docs/benchmarks/test-suite-timing.md`). **Do not** use `LyxTestHub`.

- **`internal/config` edit machinery (TDD candidate):** unit tests with a temp `_lyx/config/`
  and an **injected fake editor** (a scripted command, or an editor func, that writes known
  bytes). Cover: scaffold-when-missing (template written, then editor invoked), edit of an
  existing file, invalid-YAML → re-edit loop / abort path, unchanged-file path, and
  `_lyx`-not-initialized error from `FindBaseDir`.
- **config-CLI layer:** unit tests for the bare-`lyx config` menu (clone `ide/menu_test`
  approach — feed input via `in`, assert on `out`, validate selection/quit/out-of-range),
  and for `lyx config <module>` dispatch incl. unknown-module error. Mock/stub the sync
  trigger so menu/dispatch tests don't require a weft repo.
- **module template generators:** assert each generator (`board`, `worktree`, `weft`) produces
  parseable YAML whose keys match the module's defaults. For **board/worktree only** (these are
  relocations), assert `init` still scaffolds byte-identical content after the move (regression
  guard so the relocation is behaviour-preserving). The **weft** generator is net-new, so it has
  no identical-content baseline — assert only that it parses and yields the `pathspec` default.
- **paths `HostJunctions()` (TDD candidate):** table test asserting it returns exactly the
  `_lyx` entry with the correct link path (`HostLyxLinkHere()`) and weft target
  (`WeftLyxDir()`) for prime and non-prime worktrees. Guard test that it contains **no**
  `_codeguide` entry (scope assertion).
- **worktree seeder (integration):** after `lyx worktree add` against a `CopyPaired`
  fixture, assert the `_lyx` junction exists and points into the weft worktree and that
  `.git/info/exclude` contains `_lyx` — i.e. behaviour is unchanged after refactoring the
  seeder to iterate the list.
- **end-to-end config edit (integration):** on a paired fixture, run `lyx config <module>`
  with the fake editor, then assert the file is committed in the weft worktree (sync ran)
  and absent from the host worktree's tracked tree.
