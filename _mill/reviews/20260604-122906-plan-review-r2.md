## Findings

### [BLOCKING] Card 4 Context missing git.go
**Location:** Batch 2 (Store) / Card 4
**Issue:** Card 4 Requirements explicitly mention `atomicWrite(wikiPath, relPath, content)` with the annotation "(from `git.go`, same package)", but `internal/wiki/git.go` is absent from the Context field. Per the context-completeness rule, this is a blocking violation.
**Fix:** Add `internal/wiki/git.go` to Card 4's `Context:` list.

### [NIT] Batch 2 narrative contradicts its own DAG entry
**Location:** Batch 2 (Store) / Batch Tests section
**Issue:** The closing paragraph says "reflected in `depends-on: [1, 3]` in the DAG", but the YAML header (and overview) both correctly say `[1, 3, 5]` (batch 5 needed for `atomicWrite`).
**Fix:** Update the narrative sentence to `[1, 3, 5]`.

---

### [NIT] pathGuard spec cites wrong stdlib function; Windows path risk
**Location:** Batch 5 (Git and Lock) / Card 11
**Issue:** Requirements say "check for `..` in `filepath.SplitList`-style" — `filepath.SplitList` splits a colon/semicolon PATH env var, not path components. The OR clause offers `strings.Split(relPath, "/")`, but after `filepath.Clean` on Windows this normalises to backslashes, so a forward-slash split would miss `..` components (e.g., `..\etc\passwd` splits to one element).
**Fix:** Replace the confusing `SplitList` reference; specify `strings.HasPrefix(filepath.Clean(relPath), "..")` as the canonical check, which is correct on all platforms.

---

### [NIT] Batch Tests claims five newTask cases; only four specified
**Location:** Batch 1 (Bootstrap) / Batch Tests + Card 3
**Issue:** The Batch Tests section states "5 cases each" for `newTask`, but Card 3 Requirements enumerate only four: defaults, ID set, missing slug error, group key error.
**Fix:** Either add a fifth case (e.g., non-string slug value → error) or correct the count to four.

---

### [NIT] UpsertTasksBatch and SetDeps have no unit tests
**Location:** Batch 2 (Store) / Card 6
**Issue:** Both `UpsertTasksBatch` and `SetDeps` are implemented in Card 5 but no test cases are listed in Card 6. `UpsertTasksBatch` has its own "project then validate all, return first error" contract; `SetDeps` calls `validateWrite` and could surface cycle/dangling errors.
**Fix:** Add at least one test case each: a valid batch succeeds; a batch with one invalid task returns an error and mutates nothing.

REQUEST_CHANGES
One blocking context gap (Card 4 missing git.go) plus three low-severity nits.
