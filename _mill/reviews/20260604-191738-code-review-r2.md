## Findings

### [NIT] MergeTasks nextID off-by-one for empty projected snapshot
**Location:** `internal/wiki/store.go:269-277`
**Issue:** When `projected` is empty, the loop leaves `nextID=0` then `nextID++` yields 1, diverging from `store.nextID()` which returns 0 for an empty store. In practice this doesn't surface because the actual stored ID comes from `s.UpsertTask` (which calls `s.nextID()` on the post-removal store), and validation doesn't inspect IDs — but the `incoming` task passed to `validateWrite` carries the wrong ID.
**Fix:** Mirror `UpsertTasksBatch`'s pattern: `if len(projected) == 0 { nextID = 0 } else { maxID = ...; nextID = maxID + 1 }`.

### [NIT] PathGuard deviates from plan's cross-platform spec
**Location:** `internal/wiki/git.go:36-44`
**Issue:** Plan specifies `filepath.Clean(relPath)` + `strings.Split(cleaned, string(filepath.Separator))`. The implementation instead normalises to forward slashes before splitting, so `C:\absolute\path` on Linux passes `filepath.IsAbs` as false and the `relPath[0]=='/'` guard as false, making the test case `"absolute path on windows"` pass only on Windows.
**Fix:** Use the plan's approach: `cleaned := filepath.Clean(relPath); parts := strings.Split(cleaned, string(filepath.Separator))`.

### [NIT] Plan-specified unexported helpers are all exported
**Location:** `internal/wiki/task.go`, `layer.go`, `render.go`, `git.go`
**Issue:** Plan text names helpers as `newTask`, `applyPatch`, `computeLayers`, `renderOrder`, `extendedTitle`, `pathGuard`, `atomicWrite`, `runGit`, `pull`, `commitPush` (unexported), yet the `wiki_test` package requires exported symbols — so all are PascalCase in the implementation. This is a correct resolution of a plan inconsistency; no functional issue.
**Fix:** No action needed; noting as informational.
