## Findings

### [BLOCKING] TestMalformedYAMLError will fail after migration
**Location:** Batch 3 Card 7 + Batch 4 Card 16
**Issue:** Card 16 keeps `TestMalformedYAMLError` (assertion: error message contains `"parsing YAML"` OR `"error"`), but Card 7's `loadYAMLLayer` returns raw `yaml.v3` parse errors without wrapping. For syntactically malformed YAML (unclosed bracket / bad indentation), yaml.v3 produces scanner errors like `"yaml: line 2: did not find expected ',' or ']'"` — containing neither `"parsing YAML"` nor `"error"` as substrings. The old `board/config.go` wrapped all yaml errors with `fmt.Errorf("error parsing YAML in layer file %s: %w", ...)`, which is what made the test pass. That wrapper is not carried into `internal/config`.
**Fix:** Card 16 must update `TestMalformedYAMLError` to drop the substring assertion (keep only `err != nil`) or change it to `stringContains(errMsg, "yaml:")`, consistent with the raw error format from yaml.v3.

### [NIT] Batch-4 scope claims behaviour is unchanged; .mhgo/ layer is dropped
**Location:** Batch 4 scope description (`04-board-migration.md`, first paragraph)
**Issue:** "The external batch-4 interface … remain identical in signature and behaviour" is inaccurate — dropping the `.mhgo/<module>.yaml` second-layer is a deliberate behaviour change (correctly reflected by `TestLoad_DotMhgoIgnored` and the deletion of `TestDeepMergeMultipleLayers`), but the prose says otherwise.
**Fix:** Amend the scope sentence to note that `.mhgo/` layer support is intentionally removed as part of this simplification.
