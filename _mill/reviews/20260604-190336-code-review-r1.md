## Findings

### [BLOCKING] upsert-batch output omits required `count` field
**Location:** `cmd/mhgo/main.go:101`
**Issue:** Plan card 16 specifies `{"ok":true,"count":<n>}` for `upsert-batch`, but the implementation calls `outputSuccess()` which only emits `{"ok":true}` with no `count`.
**Fix:** After `w.UpsertTasksBatch`, output `{"ok":true,"count":len(payload.Tasks)}` inline or via a new `outputSuccessBatch(n int)` helper.

### [NIT] `NewTask` hand-codes per-field merge after JSON round-trip
**Location:** `internal/wiki/task.go:41-72`
**Issue:** The Shared Decisions section explicitly states "do not hand-code per-field merging"; `ApplyPatch` follows this correctly but `NewTask` does a full round-trip then re-overlays fields one by one with nil/zero guards.
**Fix:** Start from the default `Task`, marshal-in `fields` directly (same approach as `ApplyPatch`), then force `ID = nextID` and validate slug — the JSON zero-value for missing keys naturally preserves the pre-set defaults after an initial `json.Unmarshal` into the already-initialized struct.

### [NIT] `UpsertTasksBatch` projected-ID starts at 1 for empty store
**Location:** `internal/wiki/store.go:246`
**Issue:** The loop sets `nextID = 0`, then unconditionally does `nextID++`, so the first projected task in an empty store gets ID 1; `store.nextID()` returns 0 for an empty store (different policy). Projected IDs feed `validateWrite` labels — inconsistency is harmless today but fragile.
**Fix:** Mirror `store.nextID()`: `if len(snapshot) == 0 { nextID = 0 } else { nextID = maxID + 1 }`.

### [NIT] `wiki_test.go` uses `package wiki` while all other test files use `package wiki_test`
**Location:** `internal/wiki/wiki_test.go:1`
**Issue:** All six sibling test files are external tests (`wiki_test`); this file is an internal test. It accesses no unexported symbols, so the inconsistency is needless and the plan's Card 15 wording is ambiguous on intent.
**Fix:** Change to `package wiki_test` and import the package for consistency; or leave as-is if white-box access is anticipated. Low impact; flag for author preference.

### [NIT] Numeric `id_or_slug` from CLI JSON decoded as `float64`, not handled
**Location:** `cmd/mhgo/main.go` (get/remove/set-phase dispatch), `internal/wiki/store.go:188`
**Issue:** `encoding/json` unmarshals JSON numbers into `interface{}` as `float64`; `Store.GetTask`/`RemoveTask`/`SetPhase` switch on `int` only, so a numeric CLI argument silently returns "not found".
**Fix:** In the CLI dispatch (or in each store method), add a `case float64:` branch that casts to `int`.
