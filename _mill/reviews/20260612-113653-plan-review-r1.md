## Findings

### [NIT] Pane/MuxpocState field-name specification is ambiguous
**Location:** Batch 1, Card 1 (state.go)
**Issue:** Spec lists `type Pane struct` "JSON fields" using lowercase Go names (`id string`, `session_id string`, `kind string`); same for `MuxpocState` (`session`, `socket`, `stripped_env`, `panes`). If taken literally these would be unexported and silently fail to marshal. Cards 6–9 then reference `Pane.ID`, `Pane.SessionID`, `Pane.Kind`, `state.Session`, `state.Socket`, `state.StrippedEnv`, `state.Panes`.
**Fix:** Restate as exported Go field names with explicit JSON tags (e.g. `ID string \`json:"id"\``). Cross-card references already disambiguate, so this is precision rather than a blocker.

### [NIT] Smoke test skips on missing psmux but not on missing claude
**Location:** Batch 3, Card 13 (muxpoc_smoke_test.go)
**Issue:** Test sets `LaunchTpl = "Write-Host ready; Read-Host"` to avoid the real claude. But `cmdUp` cold-start still resolves the claude binary (`exec.LookPath("claude")` when `ClaudePath == ""`) and returns `output.Err` if absent — independent of whether the template references `%CLAUDE%`. On a CI/dev box without claude, the test fails before any psmux interaction.
**Fix:** Either set `cfg.ClaudePath = cfg.PwshPath` (or any existing binary) in step 4, or add a "skip if claude not found" guard alongside the psmux check in step 2.

### [NIT] Stripped-env contract: spec says ordered, test asserts any-order
**Location:** Batch 1, Card 1 (state.go) vs Card 2 (TestStrippedEnvKeys)
**Issue:** Card 1 says `strippedEnvKeys` returns keys "in the same order" as `sanitizeEnv` removes them. Card 2's test asserts "(in any order)". Minor inconsistency — either tighten the test or relax the spec.
**Fix:** Pick one. The order-preserving spec is more useful and the test should verify it.

### [NIT] cold-recover reports stale stripped_env
**Location:** Batch 2, Card 6 (coldRecover step 5)
**Issue:** Returns `state.StrippedEnv` from the original cold-start. If the daemon-driven recovery happens in a different process env, the reported list may not match what was actually stripped at this recover. Minor reporting fidelity issue.
**Fix:** Recompute via `strippedEnvKeys(os.Environ())` at recovery time, like cold-start does.

### [NIT] coldStart never stores real psmux pane IDs
**Location:** Batch 2, Card 6 (coldStart step 11) and Card 7 (review step 8)
**Issue:** Both store `Pane{ID: ""}`. `cmdStatus` then surfaces `state_panes` with empty IDs while `live_panes` carries real `%N` IDs — operators cannot correlate the two. The plan acknowledges this as intentional ("session rebuilt on recover"), so accept as designed; otherwise call `listPanes` post-launch to backfill `Pane.ID`.
**Fix:** Optional — backfill via `listPanes` after launching/splitting, or document the empty-ID convention in `mux.md`.

## Verdict

APPROVE
Plan is well-grounded in source patterns and design doc; findings are precision/edge-case improvements, not blockers.
