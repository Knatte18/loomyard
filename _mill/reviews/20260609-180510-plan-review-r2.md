## Findings

### [BLOCKING] Card 1 lock API description lists wrong signatures
**Location:** Batch 1 / Card 1
**Issue:** Requirements states `AcquireWriteLock(path string) (FileLock, error)`, `AcquireReadLock(path string) (FileLock, error)`, and `Release(lock FileLock) error` — but the actual `board/lock.go` returns `*FileLock` (pointer) for both Acquire functions, and `Release` is a method `(l *FileLock) Release() error`, not a free function. These wrong signatures are consistent with the misleading `Release(...)→lock.Release(...)` instructions in Cards 2, 9, 10, 11 (see below), creating a coherent but incorrect interpretation.
**Fix:** Correct the signatures to `(*FileLock, error)` for both Acquire functions and describe Release as a pointer-receiver method `func (l *FileLock) Release() error`, not a standalone function.

### [NIT] Cards 2, 9, 10, 11 describe non-existent Release(...) free-function pattern
**Location:** Batch 1 Card 2; Batch 4 Cards 9, 10, 11
**Issue:** All four cards instruct replacing `board.Release` / `Release(...)` with `lock.Release(...)`, implying Release is a free function. In the codebase it is always a method call (`defer lockVar.Release()`); no unqualified `Release(x)` call site exists to transform.
**Fix:** Remove the `Release(...)` replacement instruction; note that existing `.Release()` method calls need no change.

### [NIT] Variable `lock` shadows imported `lock` package in board.go and sync.go
**Location:** Batch 4 / Cards 9, 11
**Issue:** Cards add `import "github.com/Knatte18/mhgo/internal/lock"` but `board.go` and `sync.go` already use `lock` as a local variable name, shadowing the package identifier within each function scope.
**Fix:** Suggest an import alias (e.g., `flock`) or explicitly note the shadowing is intentional and compiles correctly in Go.

### [NIT] Card 16 suggests os.Unsetenv violating shared decision
**Location:** Batch 4 / Card 16
**Issue:** `TestLoadConfig_FallbackPathResolution` says "(use `t.Setenv` with `os.Unsetenv` or just choose a name that cannot be set in CI)" — `os.Unsetenv` mutates global OS state, which the shared decision "No global state or `os.Setenv` in tests" prohibits.
**Fix:** Remove the `os.Unsetenv` option; choosing a CI-safe name (already given as `NONEXISTENT_MHGO_TEST_VAR_XYZ`) is the correct path.

## Verdict

REQUEST_CHANGES — one blocking: Card 1's wrong lock API signatures (value vs pointer, method vs free function).
